google谷歌授权失败问题

Posted on 2021-01-29

【问题描述】登陆后进系统发送邮件正常弹出授权；刷新页面后再次发送报告报错

【问题1】为什么刷新后会报错？

异步初始化时当调用grantOfflineAccess时初始化未结束：

window.gapi.auth2.init({
        client_id: that.client_id,
        scope: "https://www.googleapis.com/auth/gmail.send"
      });
    isauth({//请求授权接口
    }).then((res) => {
      if (res.status_code == 255) {
         window.gapi.auth2.getAuthInstance().grantOfflineAccess({
        ......
        })

【如何解决】改成同步执行

isauth({ //请求授权接口
        ......
          window.gapi.auth2
            .init({
              client_id: that.client_id,
            })
            .then(() => {##控制异步操作，等待初始化执行完毕
          window.gapi.auth2.getAuthInstance().grantOfflineAccess({
                  scope: "https://www.googleapis.com/auth/gmail.send",
                })

【问题2】为什么第一次登陆后发报告不报错？

登陆时调用的插件vue-google-signin-button，其中源码中已包含了Init操作。

在项目根目录下添加.gitlab-ci.yml 并配置。介绍：

.gitlab-ci.yml 用来配置 CI 用你的项目中做哪些操作，这个文件位于仓库的**根目录**。

当有新内容push到仓库，或者有代码合并后，GitLab会查找是否有.gitlab-ci.yml文件，如果文件存在，Runners将会根据该文件的内容开始build本次commit。

.gitlab-ci.yml 使用YAML语法， 你需要格外注意缩进格式，要用空格来缩进，不能用tabs来缩进。
https://segmentfault.com/a/1190000019540360

问题1：运行ci后who am i提示用户是gitlab-runner,且执行rm命令提示无权限

解决：更改安装时设置的默认用户为root

安装gitlab-runner默认会将用户设置为：gitlab-runner，该设置会使编写.ci的脚本，操作带来很大不便，本人深受其害。

3.重新安装gitlab-runner并将用户设置为root gitlab-runner install –working-directory /home/gitlab-runner –user root

4.sudo service gitlab-runner restart重启gitlab-runner

5.再通过第一步的命令查看gitlab-runner看默认用户是否变成root

vue的跨域代理proxyTable

Posted on 2021-01-05

【问题】为什么后者设置跨域请求，有时候报错有时候不报错？

参考：https://www.cnblogs.com/wasbg/p/12659610.html

proxyTable 跨域的基本原理是:
在开发模式下,webpack 会为我们提供一个http代理服务器.
我们请求接口的时候,实际上是请求的webpack提供的这个http代理服务器.
在由这个代理服务器请求真是的数据服务器.
最后数据经由webpack代理服务器,最后转交给我们的vue程序.

为什么加个代理服务器就可以了?

因为代理服务器不是浏览器,它没有同源策略的限制

具体配置：
proxyTable: {
      // 这里配置 '/api' 就等价于 target , 你在链接里访问 /api === http://localhost:54321
      '/api': {
        target: 'http://localhost:54321/', // 真是服务器的接口地址 // http://localhost:54321/json.data.json,
        secure: true, // 如果是 https ,需要开启这个选项
        changeOrigin: true, // 是否是跨域请求?肯定是啊,不跨域就没有必要配置这个proxyTable了.
        pathRewirte: {
          // 这里是追加链接,比如真是接口里包含了 /api,就需要这样配置.

          '/^api': 'api/', 
          // 等价于 
          // step 1  /api = http://localhost:54321/
          // step 2 /^api = /api + api == http://localhost:54321/api
        }
      }
     },
proxyTable 和 axios.defaults.baseURL

前者适合本地调试使用

后者适合生产环境

【解决】为什么后者设置跨域请求，有时候报错有时候不报错？

按理说应该报错，但是若后端处理了跨域就不会报错。

http缓存分享汇总

Posted on 2020-12-22
定义

Web 缓存是指一个 Web 资源（如 html 页面，图片，js，数据等）存在于 Web 服务器和客户端（浏览器）之间的副本。
缓存会根据进来的请求保存输出内容的副本；当下一个请求来到的时候，如果是相同的 URL，缓存会根据缓存机制决定是直接使用副本响应访问请求，还是向源服务器再次发送请求。

分类

为什么使用缓存
减少网络延迟：可以明显加快页面打开速度，达到更好的用户体验。

减少网络带宽消耗：当web缓存副本被使用时，只会产生极小的网络流量，可以有效降低运营成本。

降低服务器压力：给网络资源设定有效期之后，用户可以重复使用本地缓存，减少对源服务器的请求，简介降低了对服务器的压力，同时搜索引擎的爬虫机器人也能根据过期机制降低爬取的频率，也能有效降低服务器压力。
缓存区域
from memory cache：快速读取、时效性、js/图片
from disk cache：比较慢、css文件
主要头部字段

Pragma 只有一个属性值，就是 no-cache ，效果和 Cache-Control 中的 no-cache 一致，不使用强缓存，需要与服务器验证缓存是否新鲜，在 3 个头部属性中的优先级最高。（已废弃）
Expires 的值是一个 HTTP 日期，在浏览器发起请求时，会根据系统时间和 Expires 的值进行比较，如果系统时间超过了 Expires 的值，缓存失效。
Cache-Control：
max-age：单位是秒，缓存时间计算的方式是距离发起的时间的秒数，超过间隔的秒数缓存失效
no-cache：不使用强缓存，需要与服务器验证缓存是否新鲜
no-store：禁止使用缓存（包括协商缓存），每次都向服务器请求最新的资源
private：专用于个人的缓存，中间代理、CDN 等不能缓存此响应
public：响应可以被中间代理、CDN 等缓存

有了Last-Modified为什么还需要Etag？

Last-Modified的值单位为秒。如果服务资源在1秒内发生多次变更，那服务器通过秒级对比就没办法发现文件是否已经变更。
一些特殊场景下，有些资源会被定时更新或者修改，但是其内容并未发生改变，而最后修改时间却在变化，导致客户端缓存定期失效，没有达到使用缓存的目的。
服务器可能因为一些其他原因，其时间没有与代理服务器或客户端时间保持一致，因时间偏差，从而导致缓存失效。

有了Etag为什么还需要Last-Modified？

Etag的值是服务器计算出的一串hash值，如果计算过程较复杂，比较消耗性能，那使用Last-Modified会比较合适
如果获取某些文件的最后修改时间比较容易，而文件的变更频率也不高，那无疑使用Last-Modified更合适
缓存流程

强缓存：不会向服务器发送请求，直接从缓存中读取资源，在chrome控制台的network选项中可以看到该请求返回200的状态码;

协商缓存：向服务器发送请求，服务器会根据这个请求的request header的一些参数来判断是否命中协商缓存，如果命中，则返回304状态码并带上新的response header通知浏览器从缓存中读取资源；

两者的共同点是，都是从客户端缓存中读取资源；区别是强缓存不会发请求，协商缓存会发请求。

使用

nginx配置添加响应头？

response header一般都是以key：value的形式，例如

add_header Cache-Control no-store

频繁变动的资源：

完全不缓存：no-store
协商缓存：no-cache；配合协商缓存

不经常变动的资源和文件：

css\js\图片文件资源可以设置一个较长的缓存有效期；
  只有当文件名变化的时候才更新

cache.js

let http = require('http'),
fs = require('fs'),
path = require('path'),
url = require("url")
http.createServer((req,res)=>{
  let pathname = __dirname + url.parse(req.url).pathname; // 获取文件路径
  let statusCode = 200 // 响应头状态码
  let extname = path.extname(pathname) // 获取文件扩展名
  let headType = 'text/html' // 内容类型
  if( extname){
    switch(extname){
      case '.html':
       headType = 'text/html'
      break;
      case '.js':
        headType = 'text/javascript'
      break;
    }
  }
  fs.readFile(pathname, function (err, data) {
    res.writeHead(statusCode, {
      'Content-Type': headType,
      // 'Expires':new Date(Date.now() + 3000).toUTCString()
      // 'Cache-Control':'max-age=10',
      // 'Pragma':'no-cache',

      'Cache-Control':'no-store',
      // 'Cache-Control':'no-cache'
    });
    res.end(data);
  });
}).listen(3000)
